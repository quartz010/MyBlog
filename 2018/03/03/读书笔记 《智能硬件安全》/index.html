读书笔记 《智能硬件安全》

前

这次再开一个“读本好书吧”这个系列，把平时自己对数的感受和收获记录下来，其实早都有这个想法了。现在要有实现的决心了。

关于本书

觉得未来，当互联网和软件继续向生活中渗透，之前的传统硬件慢慢的和软件融合，形成一个现在所谓的智能硬件。这个趋势是在近几年愈发的强烈，笔者是学信号的，现在SDR（Software definition Radio）也是慢慢的得到重视，可见现在软件化是一个趋势。

记得一个梗说是，以前的战斗机只是卖硬件的钱，当人们意识到软件这个东西之后，现在可能有些固件比硬件的造价都要可观。VxWorks 一个系统的授权费都是百万来着。

回到正题，所以以后硬件安全肯定是一个值得关注的地方(可以泛化到整个的安全领域)。于是乎，当自己的了解和涉猎，找了一本关于硬件安全的薄书，粗浅的学习一二。

《智能硬件安全》 刘浩建著 电子工业出版社 9787121301032

读书笔记

IoT云平台架构

自己也算是对IOT从事过简单的开发，倒也是清楚产品的架构，但是架构的分类，在本书中有讲到这里总结到。

当前IoT的云平台,大致分为三类: 转发云 功能云 第三方云. 这三类.

  1. 转发云

    主要实现云端功能用于流量的转发功能主要实现在客户端, 云端只是实现数据的转发, 实现终端到终端的通信. 实际上云端起到反向代理的作用

  2. 功能云

    是在云上实现控制及信息功能, 客户端只是实现控制发送的功能

  3. 第三方云

    这种像是, 目前流行的 物联网的统一接口网站(如 yeelink) , 这样提供了第三方的 API , 以实现,物联设备和用户终端的信息交互

目前的云产品，差不多是以上三种架构中的一直，笔者之前的产品也是属于转发云。

不过，实际上觉得，这些架构不可能是现在物联网的最终形态，link of things 万物互联，所以很难有一个强中心化的结构存在。去中心化的物联网络应该是大势所向。

IOT面临的主要安全问题

  1. 数据存储不当

    客户端，明文或者弱加密的保存敏感信息。如用户的凭证，token等。

  2. 服务端部署不当

    对客户端的输入数据未进行输入检查和标准化，可能会有用户的恶意请求。

  3. 传输过程未加密

    如请求使用Http明文，或者socket明文数据。未使用SSL/TLS。可能会有数据抓包，导致重要信息泄露。

  4. 手机客户端注入 （个人觉得应该归入服务部署不当）

  5. 身份认证措施不当

    如设备的MAC，SN 可能被枚举

  6. 敏感数据泄露

    这里指的是固件中的敏感数据，比如重要的参数和请求网站，不应以明文的方式出现在固件代码中。否则很容易被逆向得出。

APK终端安全

APK文件结构

主要的威胁出现在APK文件的逆向中。APK的结构有被编译的代码文件（。dex）、资源文件（resource）、assets、证书（certificates）、清单文件（manifest file）。

  • classes.dex 就是程序的类和逻辑生成的Dalvik虚拟机文件。
  • META-INF 是对打包文件的 SHA-1 校验。确保包的完整性。
  • lib 针对不同架构的处理器编译代码（不是在Dalvik）。一般是so动态链接库
  • assets 包含应用程序资源的目录。
  • AndroidManifest.xml 一个APK的清单文件，提供了apk的属性和参数（如权限，名字版本等）

APK反编译

  • classes.dex 反编译得到smali（相当于是dalvik的汇编）
  • .so 动态链接库（这个就直接上ida就好）

保护措施有 代码混淆（加花），应用加固（不知道是啥）

流量劫持

这里的点在于burpsuit对HTTPS流量的分析，之前觉得HTTPS相当安全，现在发现，稍有不注意，S也不能保命

https 主要是保持用户和主机间的流量不能被明文，所以当用户选择信任证书我们生成的时候就是可以抓到明文了

后

该书的后面，也讲了SDR，和部分的应用安全实例，应该作为例子参考和学习。
